# MakerRent
## Идея
Сервис аренды дорогого оборудования для хобби и мастерских между мастерами.
## Elevator Pitch
Многие мастера и инженеры-любители отказываются от своих идей, так как покупка условного 3D-принтера или лазерного гравера стоит десятки тысяч рублей. 
Для решение данной проблемы создается платформу для аренды оборудования между мастерами. В отличие от обычных досок объявлений, здесь обеспечивается верификация исправности техники и страхование каждой сделки. 
В результате владельцы получают доход от простаивающей техники, а мастера — доступ к оборудованию по цене нескольких чашек кофе.

## Роли (Целевая аудитория)
В продукте участвуют три основные роли:
1. Арендатор — ищет специфическое оборудование для временного использования.
2. Владелец — сдает свое простаивающее оборудование в аренду, чтобы получить доход.
3. Модератор/Поддержка — сотрудник платформы, решающий спорные ситуации и проверяющий пользователей.

## Формирование MVP и MLP

**Скелет MVP:**
* Добавление карточки оборудования владельцем (US 9)
* Настройка календаря (US 11)
* Поиск оборудования на карте (US 1)
* Запрос на бронь и подтверждение (US 2, 10)
* Безопасная оплата и вывод средств (US 3, 12)
* Отмена брони и блокировка юзеров (US 6, 15, 16)
* Связь с поддержкой (US 8)

**MLP (Вау-эффект и удобство):**
* Система отзывов и рейтингов (US 5, 14)
* Внутренний чат между пользователями (в MVP могут общаться по телефону) (US 4)
* Умные фильтры поиска (US 7)
* Управление видимостью объявлений (US 13)

## Детализация требований (ФТ и НФТ)

Две самые главные истории из MVP со строгими требованиями.

### История 1: Найти оборудование на карте (US 1)

**Функциональные требования (ФТ):**
* Система запрашивает доступ к геопозиции пользователя.
* Система показывает карту и область в радиусе 5 км относительно гео пользователя (изначально).
* Система грузит пины (маркеры) коворкингов/оборудования в области видимости пользователя.
* Система объединяет несколько точек в кластер (один пин с цифрой), если они находятся рядом при отдалении карты.
* Система при тапе на пин открывает системный боттом-шит (всплывающее меню снизу) с краткой информацией: фото, название станка, цена за сутки.
* Система переводит пользователя на полный экран карточки при клике на боттом-шит.

**Нефункциональные требования (НФТ):**
* Карта не лагает при зуме и скролле (плавность 60 fps).
* Пины оборудования подгружаются с бэкенда не дольше, чем за 300 мс.
* Если интернет пропадает, система показывает заглушку-уведомление «Нет соединения».
* Пины кешируются у пользователя и моментально отдаются в рамках пользования карты.

### История 2: Безопасная оплата аренды картой (US 3)

**Функциональные требования (ФТ):**
* Система при тапе на кнопку "Оплатить" открывает защищенный шлюз банка-партнера.
* Система предлагает выбор способа оплаты: СБП, банковская карта, сохраненная карта.
* Система замораживает средства на карте Арендатора (холдирование), а не переводит их напрямую Владельцу.
* Система изменяет статус бронирования на «Оплачено» в случае успешного ответа от банка.
* Система выдает алерт об ошибке и причине в случае неуспешной оплаты.
* Система автоматически отправляет чек об операции на email пользователя.

**Нефункциональные требования (НФТ):**
* Безопасность: передача платежных данных должна осуществляться только по зашифрованному протоколу HTTPS.
* Соответствие законодательству — интеграция с онлайн-кассой для автоматической выдачи чеков.
* Система платежей быстро возвращает ответ (таймаут ожидания ответа от банка не более 5 секунд).
* Удобный UI: поля для ввода карты должны иметь автоматическое форматирование (пробелы после каждых 4 цифр) и валидацию на лету.

## DDD (Domain-Driven Design): Доменные зоны

Для обеспечения слабой связности  система разделена на 4 ключевых контекста.

### Домен «Каталог и Поиск» (Catalog & Search)
**Описание:** Отвечает за хранение объявлений об оборудовании, их географические координаты, фильтрацию и отображение на карте.
*   **Глоссарий:**
    *   *Listing (Объявление)* — карточка оборудования с описанием и ценой.
    *   *Pin (Пин)* — маркер на карте.
    *   *Cluster (Кластер)* — группа пинов, объединенных при отдалении карты.
    *   *Geo-fence (Геозона)* — радиус поиска оборудования.

### Домен «Бронирование» (Booking)
**Описание:** Управляет процессом резервирования, проверкой доступности дат и жизненным циклом сделки.
*   **Глоссарий:**
    *   *Reservation (Бронь)* — запись о намерении арендовать станок.
    *   *Timeslot (Слот)* — единица времени в календаре (сутки).
    *   *Status (Статус)* — текущее состояние брони (Created, Paid, Confirmed, Cancelled).

### Домен «Финтех» (Fintech/Billing)
**Описание:** Отвечает за взаимодействие с платежным шлюзом, заморозку (холдирование) средств и выплаты владельцам.
*   **Глоссарий:**
    *   *Hold (Холдирование)* — временная блокировка суммы на карте клиента.
    *   *Payout (Выплата)* — перевод заработанных денег владельцу.
    *   *Commission (Комиссия)* — доля платформы (15%).

### Домен «Пользователи и Рейтинг» (Identity & Trust)
**Описание:** Управление профилями, верификация личностей и система отзывов.
*   **Глоссарий:**
    *   *Verification (Верификация)* — процесс подтверждения личности или исправности техники.
    *   *Renter (Арендатор)* — роль пользователя, берущего технику.
    *   *Owner (Владелец)* — роль пользователя, сдающего технику.

## BDD (Behavior-Driven Development): Сценарии

**Критический путь:** Поиск -> Выбор дат -> Оплата (Холдирование).

### Сценарий 1: Успешная оплата и создание брони
*   **Дано:** Арендатор находится на экране карточки «3D-принтер Ender 3».
*   **И:** Выбраны даты с 20.05 по 22.05 (2 суток, итого 2000 руб).
*   **Когда:** Пользователь нажимает кнопку «Оплатить», вводит данные валидной карты и подтверждает платеж.
*   **Тогда:** Платежная система успешно холдирует 2000 руб.
*   **И:** Статус брони меняется на «Оплачено, ожидает подтверждения владельцем».
*   **И:** Пользователь видит экран «Заявка успешно отправлена».

### Сценарий 2: Отказ в оплате (Недостаточно средств)
*   **Дано:** Арендатор находится на шлюзе оплаты бронирования на сумму 2000 руб.
*   **Когда:** Пользователь подтверждает оплату картой, на которой баланс 500 руб.
*   **Тогда:** Банк возвращает ошибку «Insufficient funds».
*   **И:** Система показывает пользователю уведомление: «Ошибка оплаты: недостаточно средств».
*   **И:** Пользователь остается на экране выбора способа оплаты для ввода другой карты.

### Получение списка оборудования на карте
**GET** `/v1/listings/map?lat=55.75&lon=37.61&radius=5000`

**Response (200 OK):**
```json
{
  "items": [
    {
      "id": "eq-123",
      "title": "3D-принтер Ender 3",
      "price_per_day": 1000,
      "lat": 55.752,
      "lon": 37.615,
      "image_preview": "https://cdn.makerrent.ru/img1.jpg"
    }
  ],
}
```

### Создание бронирования
**POST** `/v1/bookings`

**Request Body:**
```json
{
  "listing_id": "eq-123",
  "start_date": "2026-05-20",
  "end_date": "2026-05-21"
}
```

**Response (201 Created):**
```json
{
  "booking_id": "b-999",
  "payment_url": "https://checkout.yookassa.ru/pay/504321",
  "total_price": 2000
}
```

## Технологический стек MVP

| Компонент | Технология | Обоснование |
| :--- | :--- | :--- |
| **Backend** | **Python (FastAPI)** | Высокая скорость разработки, отличная поддержка асинхронности и авто-генерация Swagger. |
| **Database** | **PostgreSQL + PostGIS** | Надежность транзакций (ACID) для платежей + встроенные инструменты для поиска по гео-координатам. |
